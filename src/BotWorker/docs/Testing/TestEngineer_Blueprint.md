# 数字测试工程师：蓝图与工作流

## 1. 角色定义
数字测试工程师（Digital Tester）是团队中的质量守护者。它不生产业务代码，但它生产**“真相”**。

## 2. 核心能力要求 (通过提示词与技能实现)
- **理解业务逻辑**：通过阅读 `DigitalStaffService` 中的 `SystemPrompt` 和代码注释，推导预期的业务行为。
- **编写测试脚本**：能够熟练编写 xUnit 测试用例。
- **异常诊断**：当集成测试失败时，通过 AI 分析日志，指出是由于 LLM 响应不符合预期还是代码逻辑错误。

## 3. 标准工作流 (SDLC 闭环)

### 第一阶段：用例生成 (Design)
- 当“开发员工”提交新功能描述时，“测试员工”同步生成对应的测试用例（Test Cases）和 Mock 数据。

### 第二阶段：执行与监控 (Execution)
- 监控 `StaffTask` 的完成情况，一旦任务标记为 `Completed`，立即触发对应的自动化测试。

### 第三阶段：反馈与仲裁 (Arbitration)
- **合格**：更新开发员工 KPI，任务正式归档。
- **不合格**：将测试报告注入开发员工的“认知记忆”，并重新开启开发任务，标记为“Bug Fix”。

## 4. 与 C# 系统代码的集成点
- **测试执行器接口**：系统需提供一个 `ITestRunner` 接口，允许数字员工通过指令启动特定的测试集。
- **实时日志流**：将测试过程中的 `stdout` 和 `stderr` 实时推送给数字员工。

## 5. 进化方向
- 从简单的“黑盒测试”进化为“变异测试（Mutation Testing）”，主动寻找代码中的潜在逻辑漏洞。
